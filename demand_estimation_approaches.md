# Подходы к оценке спроса

Документ собирает основные варианты, которые обсуждались для восстановления истинного спроса из наблюдаемых продаж.

## Термины

- `Продано` - фактически реализованное количество.
- `Спрос` - оценка полного спроса, включая упущенные продажи.
- `is_censored` - признак того, что день, вероятно, обрезан дефицитом.
- `lost_qty` - оценка недопроданного объема в штуках.
- `last_hour` - час последней продажи за день.
- `cum_share` - накопленная доля дневного объема к конкретному часу.

## 1. Текущий подход: профиль по `Пекарня × Номенклатура`

### Идея

Для каждой пары `Пекарня × Номенклатура` строится почасовой cumulative profile:

- берутся только "полные" дни, где продажа шла до позднего часа;
- для каждого часа считается доля продаж от дневного объема;
- затем строится накопленная доля продаж по часам;
- усреднение идет по всем полным дням этой пары.

### Как используется

Если в конкретный день товар закончился рано, то:

- смотрим `last_hour`;
- берем типичную накопленную долю к этому часу;
- оцениваем спрос по формуле:

```text
Спрос = Продано / cum_share
lost_qty = Спрос - Продано
```

### Плюсы

- Хорошо учитывает специфику конкретной пекарни и конкретного товара.
- Обычно лучше, чем общий профиль на весь ассортимент.
- Подходит для восстановления SKU-level demand.

### Минусы

- Требует достаточной истории по каждой паре.
- Шумный для редких SKU.
- Может быть нестабилен, если профиль строится по малому числу дней.

### Когда использовать

- Если есть нормальная история по каждой паре.
- Если нужен наиболее точный профиль именно для SKU внутри пекарни.

## 2. Подход начальника: профиль по пекарне целиком

### Идея

Берутся все продажи пекарни на часовом таймфрейме, затем:

- для каждого часа считается доля каждой номенклатуры в общих продажах пекарни;
- далее усредняется доля по часам;
- получается профиль не по SKU, а по пекарне в целом.

### Интерпретация

Это уже не столько спрос конкретной номенклатуры, сколько общая структура продаж пекарни по времени.

### Плюсы

- Сильнее на малых данных.
- Стабильнее, если по отдельным SKU мало истории.
- Хороший fallback для слабых или редких товаров.

### Минусы

- Теряется специфика SKU.
- Смешиваются разные эффекты:
  - ассортимент;
  - stockout;
  - популярность товара;
  - временной сдвиг продаж;
  - наличие/отсутствие товара.
- Как единственный метод для оценки спроса SKU слишком грубый.

### Когда использовать

- Как уровень агрегации выше SKU.
- Как fallback, если по `Пекарня × Номенклатура` данных мало.
- Как базовый prior для иерархической схемы.

## 3. Иерархический профиль

### Идея

Сначала пробуем самый детальный уровень, а если данных недостаточно, откатываемся к более общему:

1. `Пекарня × Номенклатура`
2. `Номенклатура`
3. `Категория`
4. `Пекарня`
5. общий профиль по всем данным

### Плюсы

- Хороший баланс между точностью и устойчивостью.
- Позволяет не терять данные по редким SKU.
- Удобно внедрять как систему fallback.

### Минусы

- Нужно аккуратно выбрать правила перехода между уровнями.
- Сложнее объяснять, чем один фиксированный профиль.

### Когда использовать

- Практически всегда, если есть и массовые SKU, и редкие SKU.
- Это хороший production-подход.

## 4. Профиль по дням недели / сезону

### Идея

Профиль строится не один, а отдельно для:

- будней и выходных;
- каждого дня недели;
- праздников;
- сезонов или месяцев.

### Плюсы

- Лучше ловит различия между понедельником и субботой.
- Часто улучшает оценку спроса без сильного усложнения.

### Минусы

- Количество сегментов растет, и данных на каждый сегмент становится меньше.
- Можно легко переусложнить схему.

### Когда использовать

- Если есть заметная внутринедельная сезонность.
- Если текущий профиль плохо работает на отдельных днях.

## 5. Модель спроса только на нецензурированных днях

### Идея

Сначала выделяются дни, где не было явного stockout, и только на них обучается модель спроса.

### Плюсы

- Простая и понятная логика.
- Меньше ручных эвристик.

### Минусы

- Использует только часть данных.
- На censored-днях может недоучитывать упущенный спрос.

### Когда использовать

- Как сильный baseline.
- Если хочется минимизировать зависимость от ручных профилей.

## 6. Двухэтапная модель

### Идея

Сначала определяется вероятность censored-дня, потом оценивается потерянный объем:

- `P(censored)`
- `lost_qty | censored`

### Плюсы

- Явно отделяет факт дефицита от величины потерь.
- Может быть точнее ручной формулы.

### Минусы

- Сложнее в реализации и поддержке.
- Требует качественных меток для censored-состояния.

### Когда использовать

- Если есть цель улучшить качество именно на дефицитных днях.

## 7. Модель с цензурированным таргетом

### Идея

Не восстанавливать спрос формулой, а обучать модель, которая понимает:

- наблюдаемое `Продано` на stockout-дне - это нижняя граница спроса;
- истинный спрос выше, но неизвестен точно.

### Плюсы

- Теоретически наиболее корректный подход.
- Меньше ручных допущений.

### Минусы

- Сложнее всего внедрять.
- Не всегда удобно в LightGBM-пайплайне без дополнительной обвязки.

### Когда использовать

- Если нужен более исследовательский, но математически аккуратный вариант.

## 8. EM-подход

### Идея

Итеративная схема:

1. грубо восстановили `Спрос`;
2. обучили модель;
3. переоценили censored-дни уже с учетом модели;
4. повторили несколько раз.

### Плюсы

- Может совместить профиль, календарные признаки и историю продаж.

### Минусы

- Сложнее контролировать.
- Есть риск переобучения на собственные ошибки.

### Когда использовать

- Если хочется исследовать более продвинутую iterative estimation схему.

## Практическая оценка

Если сравнивать по полезности для проекта:

1. `Пекарня × Номенклатура` профиль - лучший базовый метод для SKU-level demand.
2. Иерархический профиль - лучший компромисс между качеством и устойчивостью.
3. Профиль по пекарне целиком - хороший fallback, но не замена SKU-подходу.
4. Двухэтапная модель - перспективна, если хочется поднять качество на censored-днях.
5. Цензурированная модель - наиболее научный вариант, но самый сложный.

## Рекомендация

Для текущего проекта разумный порядок такой:

1. оставить текущий профиль по `Пекарня × Номенклатура`;
2. добавить fallback по пекарне целиком;
3. сделать иерархию профилей;
4. отдельно проверить двухэтапную модель восстановления спроса.

Это даст хороший баланс между качеством, простотой и воспроизводимостью.
