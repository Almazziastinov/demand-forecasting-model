# Project Status

## Назначение этого файла

Этот документ фиксирует текущую схему проекта, чтобы в репозитории было однозначно понятно:

- что является основным исследовательским контуром;
- что считается legacy;
- что существует только как demo;
- какие артефакты трактовать как рабочие внутри текущей исследовательской фазы.

## 1. Current Research Path

Текущий основной контур: `src/experiments_v2/`

Это зона, где:

- проверяются новые гипотезы;
- сравниваются архитектуры и таргеты;
- появляются новые фичи и схемы обучения;
- формируются кандидаты на следующий сильный baseline.

Именно этот контур следует считать главным при вопросах:

- куда добавлять новый эксперимент;
- где смотреть последние идеи;
- что считать актуальной исследовательской логикой.

Сопутствующие артефакты:

- `src/experiments_v2/*/run.py`
- `src/experiments_v2/*/metrics.json`
- `src/experiments_v2/*/predictions.csv`
- `src/experiments_v2/EXPERIMENTS.md`

## 2. Legacy Baseline

Legacy-ветка:

- `src/preprocessing.py`
- `src/models/train_and_save.py`
- `run_pipeline.py`
- часть тестов, завязанных на эту схему

Эта ветка полезна как:

- исторический baseline;
- быстрый контрольный сценарий;
- reference point для сравнения с новыми подходами.

Эту ветку не нужно трактовать как текущую каноническую архитектуру проекта.

## 3. Demo Web

Папка `web/` имеет статус demo-only.

Это означает:

- web-часть нужна для визуальной проверки сценариев;
- она не обязана совпадать со структурой текущего исследовательского пайплайна;
- она не должна использоваться как аргумент, что исследовательский код "обязан" быть организован точно так же;
- её допустимо развивать локально под демонстрационные задачи без полной синхронизации с `experiments_v2`.

## 4. Tests

Текущее состояние тестов следует трактовать прагматично:

- тесты проверяют не весь исследовательский контур;
- часть тестов относится к legacy-пайплайну;
- тесты для `web/` не задают архитектурную истину проекта.

Если потребуется усиление тестового покрытия, его стоит строить вокруг:

- инфраструктурных инвариантов;
- выбранного текущего baseline;
- отдельных критичных утилит и преобразований данных.

## 5. Каноничность артефактов

На текущем этапе не требуется единый "production artifact contract" для всего репозитория.

Практическое правило:

- лучшие исследовательские результаты искать в `src/experiments_v2/`
- legacy-модель и meta в `models/` считать baseline-артефактами
- web-артефакты считать demo-специфичными

## 6. Правило для новых изменений

При добавлении новых изменений использовать следующую логику:

1. Если это новая ML-гипотеза, писать в `src/experiments_v2/`
2. Если это поддержка исторического baseline, править legacy pipeline
3. Если это только визуализация или сценарий показа, править `web/`
4. Не пытаться насильно унифицировать все три контура без отдельной причины

## 7. Следующий разумный шаг

Следующий организационный шаг для проекта:

- поддерживать `experiments_v2` как главный исследовательский трек;
- не раздувать legacy-ветку;
- держать demo web отдельно;
- при необходимости постепенно формировать новый baseline из лучшего эксперимента, а не из старого pipeline по умолчанию.
