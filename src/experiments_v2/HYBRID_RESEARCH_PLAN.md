# План исследования гибридной модели

Этот документ фиксирует исследовательское направление для гибридной системы прогнозирования спроса в `experiments_v2`.

## Контекст

Глобальная модель в этой задаче не является достаточной сама по себе.

Причины:

- поведение SKU сильно отличается между пекарнями и продуктами;
- часть SKU стабильно предсказуема;
- часть SKU имеет редкий, прерывистый или шумный спрос;
- разные модели выигрывают на разных SKU.

Следовательно, цель исследования не в том, чтобы найти одну универсальную модель, а в том, чтобы построить систему, которая:

- оценивает предсказуемость SKU;
- маршрутизирует SKU к подходящей модели;
- сохраняет безопасный fallback для слабых или шумных рядов.

## Основная гипотеза

Гибридная система будет устойчиво лучше любой одиночной модели, если выбор модели строится на уровне SKU-поведения, а не только на глобальных метриках.

Это означает:

- глобальные метрики остаются ориентиром;
- но выбор модели должен опираться на режим SKU, стабильность ряда и локальный backtest;
- одного leaderboard-ранга недостаточно.

## Почему важен `R2 < 0`

SKU-пары с устойчиво отрицательным `R2` часто сигнализируют о том, что:

- ряд слишком шумный для текущего набора признаков;
- модель не учитывает ключевые драйверы;
- или для этой SKU-пары лучше работает простой fallback, чем сложный learner.

Важно:

- нельзя отбрасывать SKU-пары по финальному test split напрямую;
- нужно использовать rolling backtest или исторические окна;
- иначе критерий отбора будет использовать информацию из теста.

Практически это означает, что лучше не “выкидывать плохие SKU”, а:

- переводить их в fallback;
- снижать их вес при обучении;
- либо обучать для них отдельный класс моделей.

## Исследовательские вопросы

1. Какие SKU-пары стабильно предсказуемы?
2. Какие SKU-пары стабильно плохо прогнозируются?
3. Улучшится ли глобальная модель, если слабые SKU удалить или понизить им вес?
4. Можно ли предсказать выбор лучшей модели по meta-features SKU?
5. Какие семейства моделей лучше работают в разных режимах спроса?
6. Даёт ли иерархический ensemble выигрыш над плоским гибридом?

## Предлагаемые эксперименты

### 1. Карта предсказуемости SKU

Цель:

- разделить SKU-пары на `stable`, `borderline` и `unpredictable`.

Метод:

- запуск rolling backtest;
- расчёт `R2`, `MAE`, `WMAPE`;
- измерение доли окон с `R2 < 0`;
- измерение стабильности выбора лучшей модели по окнам.

Ожидаемый результат:

- карта того, какие SKU-пары вообще можно стабильно прогнозировать.

### 2. Исключение слабых SKU против downweighting

Цель:

- проверить, вредят ли слабые SKU глобальной модели.

Варианты:

- обучать на всех SKU-парах;
- исключать стабильно слабые SKU;
- уменьшать вес стабильно слабых SKU;
- сохранять все SKU, но переводить слабые ряды в fallback.

Сравнивать:

- глобальный `MAE`;
- глобальный `WMAPE`;
- глобальный `R2`;
- coverage.

Ожидаемый результат:

- понятная политика по слабым SKU-парам.

### 3. Meta-model / router

Цель:

- предсказывать, какую модель нужно использовать для конкретной SKU-пары, по её историческим признакам.

Кандидатные meta-features:

- средний спрос;
- доля нулей;
- коэффициент вариации;
- доля censored дней;
- тренд;
- длина истории;
- частота stockout;
- кластер пекарни;
- кластер SKU;
- категория.

Сравнивать:

- oracle best-per-SKU;
- rule-based router;
- learned router.

Ожидаемый результат:

- гибридный селектор, пригодный для production.

### 4. Сегментация SKU по режимам спроса

Цель:

- группировать SKU-пары не по модели, а по типу поведения.

Возможные режимы:

- стабильный;
- sparse;
- intermittent;
- seasonal;
- volatile;
- high-censoring.

Далее сравнивать семейства моделей внутри каждого режима.

Ожидаемый результат:

- интерпретируемая карта “какая модель для какого типа SKU”.

### 5. Иерархический ensemble

Цель:

- проверить, может ли каскадная архитектура превзойти плоский best-model выбор.

Возможная схема:

- global model для общего паттерна;
- local adjustment для эффектов пекарни или SKU;
- fallback baseline для слабых SKU.

Ожидаемый результат:

- архитектура, которая сохраняет и качество, и coverage.

### 6. Анализ ошибок

Цель:

- понять, почему именно возникают ошибки.

Разрезы:

- пекарня;
- категория;
- stockout;
- изменение ассортимента;
- новые товары;
- sparse demand;
- высокая волатильность.

Ожидаемый результат:

- конкретные причины провалов модели и список недостающих факторов.

### 7. Бейзлайны для sparse / intermittent SKU

Цель:

- проверить, не лучше ли для редких рядов простые методы.

Сравнивать:

- moving average;
- two-week average;
- Croston-like baselines;
- двухэтапные схемы;
- zero-inflated подходы.

Ожидаемый результат:

- отдельная fallback-стратегия для sparse SKU-пар.

## Что измерять

Для каждого эксперимента фиксировать:

- `R2`;
- `MAE`;
- `MSE`;
- `WMAPE`;
- coverage;
- долю SKU-пар с положительным `R2`;
- стабильность по backtest-окнам;
- win count по SKU-парам.

Для router-экспериментов дополнительно фиксировать:

- accuracy выбора модели;
- gap до oracle;
- распределение использования моделей;
- coverage слабых SKU.

## Рекомендуемый порядок работ

1. Построить карту предсказуемости SKU.
2. Проверить исключение, downweighting и fallback для SKU с `R2 < 0`.
3. Построить router по meta-features.
4. Сегментировать SKU по режимам спроса.
5. Проверить иерархический ensemble.
6. Провести анализ ошибок.
7. Добавить отдельные бейзлайны для sparse-demand SKU.

## Практическая интерпретация

Ключевой сдвиг здесь такой:

- не пытаться заставить все SKU жить в одной глобальной модели;
- не удалять слабые SKU без анализа;
- а сначала классифицировать их и затем маршрутизировать.

Это даёт систему, которая:

- исследовательски прозрачна;
- и одновременно пригодна для production.

## Ожидаемый итог

Финальный результат исследования должен отвечать на три вопроса:

- какая модель нужна для каждой SKU-пары;
- когда fallback лучше сложной модели;
- и насколько выигрывает маршрутизация по сравнению с попыткой использовать одну модель для всех.
